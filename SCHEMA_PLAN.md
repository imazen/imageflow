# Schema Versioning, Enforcement, and Codegen Plan

Status: design draft. Not yet implemented.

This document describes the plan for evolving the zennode schema system to
support multi-version nodes, mechanically-enforced compatibility, and
multi-language code generation for downstream client SDKs.

## Goals

1. **Single source of truth** per `(node_id, major_version)` pair: the Rust
   struct annotated with `#[derive(Node)]` in the crate that owns it.
2. **Breaking changes fail the build** — narrowing a numeric range, removing
   a parameter, removing an enum variant, or otherwise violating wire-level
   backward compatibility within an existing major version causes CI to fail.
3. **Additive changes require explicit snapshot refresh** but do not break
   builds, with the refresh step itself running compatibility checks so it
   cannot silently accept breaking changes.
4. **Multi-language code generation** derives every language client
   (Rust, C#, TypeScript, Go, Python) from the same schema artifacts with
   coordinated version pinning via named profiles.
5. **Clients that only generate wire format** (JSON, RIAPI querystrings, CLI
   argv, HTTP bodies) pay zero compile-time cost for the execution engine.
   Only clients that explicitly want in-process execution pull in the full
   zen crate tree.

### Separate repos (unchanged)

- **zennode** — schema traits, derive macro, registry, snapshot compat
  checker (`zennode::snapshot` module). Stable foundation.
- **Codec implementation crates** — `zenjpeg`, `zenpng`, `zenwebp`,
  `zengif`, `zenavif`, `zenjxl`, `zentiff`, `zenbitmaps`, `heic`, etc.
  Independent release cadences matching upstream libraries.
- **Foundation crates** — `zenpixels`, `zenpixels-convert`, `linear-srgb`,
  `zencodec`, `enough`, `whereat`, `archmage`, `magetypes`. Stable, rarely
  touched.
- **Pipeline utility crates** — `zenresize`, `zenlayout`, `zenblend`,
  `zenquant`. Published independently, depended on by zenpipe via version
  deps.
- **Language client repos** — `imageflow-dotnet`, `imageflow-node`,
  `imageflow-go`, `imageflow-python`. Each regenerates code from profile
  artifacts published by the monorepo.

### Why this split

- `zenpipe`, `zenfilters`, `zencodecs` release together frequently and
  their schemas feed the generated code in `imageflow-api` and `imageflow`.
  Keeping them in one workspace enables atomic commits across schema +
  wire types + translation + engine.
- `zenresize`, `zenlayout`, `zenblend`, `zenquant` are lower-level
  pipeline components. Their APIs are consumed by zenpipe but their
  release cadence is slower and independent. They don't define
  user-facing node schemas that feed imageflow-api's wire types.
- Codec crates match upstream library cadences (mozjpeg releases, jpeg-xl
  versions, etc.) and should not be coupled to the monorepo's release.

## Crate Layering and Dependency Rules

### The four imageflow crates

```text
zennode  (foundation — schema traits, registry, derive macro)
   │
   ├── zen foundation crates (zenpixels, linear-srgb, zencodec, ...)
   │
   ├── zen pipeline crates (zenresize, zenlayout, zenblend, zenquant)
   │
   ├── codec crates (zenjpeg, zenpng, zenwebp, ...)
   │        │
   │        ▼
   ├── zencodecs + zenfilters
   │        │
   │        ▼
   │   zenpipe (execution engine)
   │        ▲
   │        │
   │   imageflow-api (wire types, dispatch, Executor trait — NO zen deps)
   │        ▲
   │        │
   │   imageflow (bundle: re-exports imageflow-api, provides
   │    │         InProcessExecutor, depends on zenpipe + zen crates)
   │    │
   │    ▼
   imageflow-abi (C FFI, cdylib — depends only on imageflow)
```

**Dependency rules (invariants):**

- `imageflow-api` has **zero** dependencies on any zen crate (not zenpipe,
  not zennode, not zenfilters, not zencodecs). It is a pure wire-format
  crate with deps on `serde`, `serde_json`, `thiserror`, and nothing else.
  This is what makes it fast to compile for wire-only consumers.
- `zenpipe` has **zero** dependencies on `imageflow-api` or any imageflow
  crate. It is a pure execution engine that accepts
  `Vec<Box<dyn NodeInstance>>` and produces bytes. No wire format knowledge.
- `imageflow` (the bundle) depends on **both** `imageflow-api` and
  `zenpipe` (plus `zencodecs`, `zenfilters`, and `zennode`). It is the
  **only** place where wire types and execution types meet. Translation
  code lives here.
- `imageflow-abi` depends only on `imageflow` (the bundle). It forwards
  JSON to `Engine::send_json`.
- **No circular dependencies are possible** because `imageflow-api` and
  `zenpipe` never import each other. The bundle sits above both.

### Executor trait (injected, not feature-gated)

`imageflow-api` defines the `Executor` trait but has **no** `in-process`
feature flag. The concrete `InProcessExecutor` lives in the `imageflow`
bundle crate, not in `imageflow-api`. This eliminates feature-flag
complexity and makes the dependency DAG statically verifiable.

```rust
// imageflow-api/src/executor.rs
pub trait Executor {
    fn execute(&self, req: ExecuteRequest<'_>) -> Result<ExecuteOutput, ExecuteError>;
    fn probe(&self, bytes: &[u8]) -> Result<ProbeInfo, ExecuteError>;
    fn capabilities(&self) -> Capabilities;
}
```

```rust
// imageflow-api/src/dispatch.rs
pub fn dispatch_json(
    method: &str,
    json: &[u8],
    executor: &dyn Executor,
) -> JsonResponse { ... }
```

Endpoint dispatch (routing `v1/build`, `v3/build`, etc.) lives in
`imageflow-api` because it is a wire-format concern. The executor is
injected by the caller.

```rust
// imageflow/src/engine.rs (the bundle crate)
pub struct Engine { executor: InProcessExecutor }

impl Engine {
    pub fn new() -> Result<Self, InitError> { ... }
    pub fn send_json(&self, method: &str, json: &[u8]) -> JsonResponse {
        imageflow_api::dispatch_json(method, json, &self.executor)
    }
}
```

### Wire → NodeInstance translation

Translation from `imageflow_api::NodeV3` to `Box<dyn zennode::NodeInstance>`
happens in the `imageflow` bundle crate via generated code:

```rust
// imageflow/src/generated/translate.rs (GENERATED by zen-codegen)
pub fn node_to_instance(node: &imageflow_api::NodeV3) -> Box<dyn zennode::NodeInstance> {
    match node {
        NodeV3::Contrast(c) => Box::new(zenfilters::filters::contrast::v2::Contrast {
            amount: c.amount(),
            pivot:  c.pivot(),
        }),
        NodeV3::EncodeJpeg(e) => Box::new(zencodecs::encode_jpeg::v1::EncodeJpeg {
            quality: e.quality(),
            // ...
        }),
        // ... one arm per (node_id, major) in the profile
    }
}
```

This file is regenerated by `cargo xtask regenerate` alongside the
`imageflow-api/src/generated/` files. Both are committed and both come
from the same profile snapshot.

## Two-Dimensional Versioning

Schema versioning uses a `(major, minor)` pair per node:

- **Major version** = a separate Rust type in a separate module. Breaking
  changes create new major versions. Old majors stay registered and runnable
  for as long as the profile supports them.
- **Minor version** = additive evolution within a single Rust type. Tracked
  by `#[param(since_minor = N)]` attributes on individually-added parameters.

**Rule: minor = same struct, major = new struct.** Never conflate the two.

### Source of truth: the Rust struct

Each node's canonical definition is a `#[derive(Node)]` struct in a versioned
module within its owning crate. Example layout:

```text
zenfilters/src/filters/contrast/
├── mod.rs      // pub use v2::Contrast; plus pub mod v1, pub mod v2
├── v1.rs       // frozen — old algo, kept for reproducibility
└── v2.rs       // current — active development, minor version bumps here
```

```rust
// zenfilters/src/filters/contrast/v2.rs
#[derive(Node, Clone, Debug, Default)]
#[node(id = "zenfilters.contrast", major = 2, minor = 3)]
#[node(group = Tone, role = Filter, coalesce = "fused_adjust")]
#[non_exhaustive]
pub struct Contrast {
    /// Contrast strength (positive = increase, negative = flatten).
    #[param(range(-1.0..=1.0), default = 0.0, identity = 0.0, step = 0.05)]
    #[param(slider = SquareFromSlider, unit = "", section = "Main")]
    #[kv("s.contrast")]
    pub amount: f32,

    /// Pivot point for the power curve. Added in 2.1.
    #[param(range(0.0..=1.0), default = 0.5691, step = 0.01, since_minor = 1)]
    #[param(section = "Advanced")]
    pub pivot: f32,
}

impl Filter for Contrast { /* SIMD power curve */ }
```

The derive macro captures all metadata into a static `&'static NodeSchema`.
JSON Schema, language clients, and documentation are all derived from this
single declaration.

### Schema fields for the two-dimensional model

```rust
pub struct NodeSchema {
    pub id: &'static str,
    pub major: u32,               // NEW: Rust-type-level versioning
    pub minor: u32,               // NEW: additive-evolution versioning
    pub compat_minor: u32,        // oldest minor this code can deserialize
    // ... other existing fields
}

pub struct ParamDesc {
    pub since_minor: u32,         // NEW: minor in which this param was added
                                  //      (meaningful only within one major)
    // ...
}
```

The existing `version: u32` and `compat_version: u32` fields are superseded
by `major`/`minor`/`compat_minor`. A compatibility shim keeps `version()` and
`compat_version()` as method accessors during the transition.

## Enforcement Mechanism

The enforcement strategy is **per-major snapshot files with additive-compat
checking**. Because `zenpipe`, `zenfilters`, and `zencodecs` are all in the
same workspace, the snapshot enforcement is a single layer: workspace-wide
snapshot tests that cover every registered `(node_id, major)`.

### Snapshot files

The monorepo commits a snapshot file per `(node_id, major)` in a shared
`schemas/` directory:

```text
schemas/
├── zenfilters.contrast/
│   ├── v1.json       # frozen — any change fails CI
│   └── v2.json       # updates allowed only for additive changes
├── zenfilters.exposure/
│   └── v1.json
├── zencodecs.encode_jpeg/
│   └── v1.json
├── zenpipe.composite/
│   └── v1.json
└── zenresize.constrain/
    └── v1.json
```

Each snapshot is the full JSON Schema for that `(node_id, major)` pair, as
emitted by `zennode::json_schema::node_to_json_schema`.

### Snapshot tests

A workspace-level test walks `full_registry()` and, for every
`(id, major)` with a committed snapshot, runs the additive-compat checker:

```rust
// tests/schema_snapshots.rs (workspace-level integration test)

#[test]
fn all_snapshots_match() {
    let registry = zenpipe::full_registry();
    for def in registry.all() {
        let schema = def.schema();
        let path = format!("schemas/{}/v{}.json", schema.id, schema.major);
        if let Ok(committed) = std::fs::read_to_string(&path) {
            zennode::snapshot::assert_snapshot_matches(schema, &committed);
        } else {
            panic!("missing snapshot file: {path}");
        }
    }
}
```

Behavior of `assert_snapshot_matches`:

- **Exact match** — test passes silently.
- **Additive-only diff** — test fails with instruction to run
  `cargo xtask snapshot` to refresh. This forces drift to be committed
  explicitly.
- **Breaking diff** — test fails with a detailed `CompatReport`, listing
  each breaking change with field names and old/new values, plus the
  instruction to either revert the change or create a new major version.

Because all crates are in the same workspace, there is no separate
"imported snapshots" layer. The `schemas/` directory is the single source
of committed schema state for the entire workspace.

### Enforcement for external dep crates

Crates outside the monorepo that define node schemas (`zenresize`,
`zenlayout`, etc.) also have their snapshots committed to the monorepo's
`schemas/` directory. When an external dep updates (e.g., `zenresize`
bumps and changes the Constrain schema), the workspace snapshot test
fails until `cargo xtask snapshot` is run to refresh. The xtask runs
`check_additive_compat` and refuses to accept breaking changes.

## Additive Compatibility Rules

The `zennode::snapshot::check_additive_compat` function applies the
following rules:

| Change | Verdict | Version bump |
|---|---|---|
| Add optional parameter (`Option<T>` or with default) | additive | minor |
| Add required parameter | **breaking** | new major |
| Remove any parameter | **breaking** | new major |
| Rename parameter with `#[param(json_alias = "old_name")]` | additive | minor |
| Rename parameter without alias | **breaking** | new major |
| Change parameter kind (f32 → i32, String → Enum) | **breaking** | new major |
| Numeric `min` decreases or `max` increases (widening) | additive | minor |
| Numeric `min` increases or `max` decreases (narrowing) | **breaking** | new major |
| Enum variant added | additive | minor |
| Enum variant removed | **breaking** | new major |
| KV alias added | additive | minor |
| KV key removed | **breaking** | new major |
| Default value changed | warning | minor with warning |
| Identity value changed | warning | minor with warning |
| Step value changed | silent | minor |
| Unit / label / section / description changed | silent | minor |
| `since_minor` added to new param | additive | minor |
| `compat_minor` increased | **breaking** (drops support for older minors) | evaluated separately |
| `major` mismatched with committed snapshot | **breaking** (always) | new major |

The `CompatReport` returned by `check_additive_compat` lists all detected
changes, categorized into `breaking: Vec<BreakingChange>` and
`additive: Vec<AdditiveChange>`. A breaking change list with any entries
blocks the build; an additive-only change list prompts snapshot refresh.

## Wire Format and Profiles

### Wire format versioning

The wire envelope carries an explicit API version that maps to a set of
major versions per node:

```json
{
  "api_version": "zen-2026-q2",
  "framewise": {
    "steps": [
      {"decode":      {"io_id": 0}},
      {"constrain":   {"w": 800, "mode": "fit_crop"}},
      {"contrast":    {"amount": 0.5}},
      {"contrast@1":  {"amount": 0.5}},
      {"encode_jpeg": {"quality": 85}}
    ]
  }
}
```

Resolution rules:

1. **Unpinned node name** (`"contrast"`): look up in the active profile. If
   present, use that major. Otherwise, fall through to `latest_major()` (or
   fail if the profile is in strict mode).
2. **`@N` suffix** (`"contrast@1"`): use major N exactly. Fail if the
   registry does not contain a `NodeDef` for that `(id, major)` pair.
3. **`api_version` omitted**: use the library's compiled-in default profile.

### API profiles

A profile is a named coordinate describing:

- Major version of every node it exposes
- Whether unlisted nodes fall through to `latest_major()` or are rejected

```toml
# profiles/zen-2026-q2.toml
name = "zen-2026-q2"
description = "Q2 2026 profile — adds contrast v2 and exposure v2"

[nodes]
"zenfilters.contrast"   = 2
"zenfilters.exposure"   = 2
"zenresize.constrain"   = 1
"zencodecs.encode_jpeg" = 1
# ... every node explicitly listed

[resolution]
fallback = "latest_major"     # or "strict" to reject unlisted nodes
```

Profiles are immutable once released. New profile = new name.

### Version mismatch validation

When a client built against a newer profile sends JSON to an older server:

1. **Strict validation** (default on v3 endpoints) rejects unknown fields
   and unknown node types with structured error messages. This is the
   primary correctness guarantee — "does it parse in Strict mode" is
   sufficient to ensure the server can execute the request correctly.
2. **`imageflow_abi_profile_version()`** — a C ABI function that returns
   the profile string the native binary was built against. Language wrappers
   call this at startup for an early mismatch warning.
3. **`api_version` in responses** — the server echoes its profile version
   in every JSON response, allowing clients to detect skew.

The reverse direction (older client, newer server) is always safe — the
server has defaults for parameters the client didn't send.

### Registry lookup

The registry index becomes `(id, major)`:

```rust
impl NodeRegistry {
    pub fn get_versioned(&self, id: &str, major: u32) -> Option<&'static dyn NodeDef>;
    pub fn latest_major(&self, id: &str) -> Option<u32>;
    pub fn get_with_profile(&self, id: &str, profile: &Profile) -> Option<&'static dyn NodeDef>;
}
```

## Generated Wire Types and Validation

### Why generated, not re-exported

`imageflow-api` generates its own Rust wire types from the schema instead
of re-exporting the zen crate structs directly. This is necessary because:

- `imageflow-api` must compile without any zen crate deps (for fast
  wire-only builds).
- The zen crate structs (`zennode_defs::Contrast`, etc.) have no validation
  — they're public fields with no range enforcement.
- String-typed params (`mode: String`) should be typed enums
  (`mode: ConstraintMode`) in the public API.
- Serde alone doesn't enforce zennode's schema constraints (ranges, enum
  validity, unknown-field policy).

### Schema-driven validating parsers

`zen-codegen` reads the schema and emits a custom `parse()` function per
wire type. The parser validates every constraint the schema describes:

- Numeric range enforcement (errors or clamps depending on mode)
- Typed enum variant checking (compile-time, no string typos possible)
- Unknown-field rejection (in Strict mode) or warning (in Lenient mode)
- Default value injection for missing optional fields
- `NaN`/`Infinity` rejection for float fields

Validation mode is set per endpoint:

```rust
pub enum ValidationMode {
    /// Reject unknown fields, out-of-range values, missing required fields.
    Strict,
    /// Clamp out-of-range, warn on unknown fields.
    Lenient,
    /// Match imageflow v2's historical quirks for backward compatibility.
    LegacyV2,
}
```

v1 endpoints use `LegacyV2`. v3 endpoints use `Strict`. Developer tools
may use `Lenient`.

## Multi-Language Code Generation

### Architecture

`zen-codegen` is a Rust crate in `tools/zen-codegen/` that reads JSON
Schema (with zennode's `x-zennode-*` extensions) and emits source files
for target languages:

```rust
pub trait Target {
    fn name(&self) -> &'static str;
    fn generate(&self, input: &CodegenInput) -> GeneratedFiles;
}

pub mod rust;        // wire structs + validating parsers + NodeV3 enum
pub mod csharp;      // classes + [DataContract] + XML docs
pub mod typescript;  // interfaces + runtime validation
pub mod go;          // structs + json tags + methods
pub mod python;      // Pydantic models
```

The existing `zenpipe::codegen_csharp` logic is ported as the initial
`csharp` target.

### Two codegen outputs for Rust

The same profile input produces two distinct sets of generated files within
the monorepo:

1. **`imageflow-api/src/generated/`** — wire types with validating parsers,
   typed enums, `NodeV3` enum. No zen deps.
2. **`imageflow/src/generated/translate.rs`** — `NodeV3 → Box<dyn NodeInstance>`
   match. Depends on zen crates.

Both are regenerated atomically by `cargo xtask regenerate` and committed
together. Since they're in the same workspace, there is no version
coordination problem — they always come from the same commit.

### Language client repos

Each language client (`imageflow-dotnet`, `imageflow-node`, etc.) lives in
its own repo and consumes:

- Profile artifacts from the monorepo (aggregated JSON schemas)
- Native binaries from the monorepo (imageflow.dll / libimageflow.so)
- zen-codegen for its target language

When the monorepo tags a release, language client CI triggers and
regenerates + publishes.

For .NET specifically, the managed package (`Imageflow.Net`) and native
runtime packages (`Imageflow.NativeRuntime.*`) are separate NuGet packages.
Version matching is enforced by:

- NuGet version constraints between managed and native packages
- A runtime `imageflow_abi_profile_version()` check at startup

## Publishing

### Workspace path deps → crates.io version deps

All workspace members use `path + version` in their Cargo.toml:

```toml
# imageflow/Cargo.toml
[dependencies]
imageflow-api = { path = "../imageflow-api", version = "3.1.2" }
zencodecs     = { path = "../zencodecs",     version = "0.4.0" }
zenfilters    = { path = "../zenfilters",     version = "0.3.0" }
zenpipe       = { path = "../zenpipe",        version = "0.8.0" }
```

Local builds use the path. `cargo publish` strips the path and keeps the
version. All workspace crates publish from the same commit.

### Publish xtask

`cargo xtask publish` handles the full release:

1. **Preflight**: detect crates with changes since the last release tag.
   Error if any changed crate hasn't had its version bumped.
2. **Publish**: topological order, wait for crates.io indexing between each.
   Order: zencodecs → zenfilters → zenpipe → imageflow-api → imageflow.
   (`imageflow-abi` is not published to crates.io — it ships as binary
   artifacts.)
3. **Clean-room verify**: create a throwaway project that depends on the
   just-published `imageflow` from crates.io (no path overrides), build
   and test it. Catches stale or forgotten publishes.
4. **Tag**: `git tag release-YYYY-MM-DD`.
5. **Build artifacts**: cross-compile `imageflow-abi` for all target
   platforms, upload binaries to GitHub release.

### Version alignment

Because all crates are in the same workspace and publish from the same
commit, there is no version pin problem. Users who depend on
`imageflow = "3.1"` get a version of `imageflow-api` that was published
at the same time from the same source tree.

Users who depend on `imageflow-api` alone (wire-only) get the standalone
crate with zero zen deps. Users who depend on `imageflow` get the full
stack (engine + wire + executor) with `imageflow-api` re-exported.

## Versioning Scheme

### Crate versions

SemVer at the Rust API level. Adding a new node-major is a minor crate
bump. Removing an old node-major is a major crate bump.

All workspace crates share a release cadence but have independent version
numbers (zencodecs 0.4.0, zenfilters 0.3.0, imageflow-api 3.1.2, etc.).

### Profile versions

Named, immutable once released. Pin per-node majors. New profile = new
name; never edit old ones. Profiles live in `profiles/` in the monorepo.

### Example trajectory

| Event | Workspace crates | Profile | Published |

## Worked Examples

### Scenario A: add an optional parameter

Developer adds `target_butteraugli: Option<f32>` to `EncodeJxl`:

1. Edit `zencodecs/src/zennode_defs.rs` (or `zencodecs/src/nodes/encode_jxl/v1.rs`
   after the versioned-module refactor). Add the field with
   `#[param(range(0.0..=25.0), since_minor = 1)]`.
2. `cargo xtask snapshot` — updates `schemas/zencodecs.encode_jxl/v1.json`.
3. `cargo xtask regenerate` — updates both `imageflow-api/src/generated/`
   (new wire field + validator) and `imageflow/src/generated/translate.rs`
   (new field copy in the match arm).
4. `cargo test --workspace` — all snapshot tests, wire golden tests, and
   integration tests pass.
5. Commit all changed files in one PR. CI green. Self-merge.
6. `cargo xtask publish` — publishes everything in order, runs clean-room
   verify, tags release.

**One PR. One CI run. ~30 minutes wall-clock.**

### Scenario B: remove a parameter (breaking change)

Developer removes `deringing` from `EncodeJpeg`:

1. Attempt to delete the field from v1. `cargo test` fails:
   *"BREAKING: zencodecs.encode_jpeg v1 param 'deringing' removed."*
2. Create `zencodecs/src/nodes/encode_jpeg/v2.rs` without `deringing`.
   Keep v1 unchanged.
3. `cargo xtask snapshot zencodecs.encode_jpeg@2 --new-major` — creates
   `schemas/zencodecs.encode_jpeg/v2.json`.
4. Create new profile `profiles/zen-2026-q3.toml` pinning encode_jpeg at
   major 2.
5. `cargo xtask regenerate` — `imageflow-api` now generates `EncodeJpeg`
   (v2, no deringing), `EncodeJpegV1` (explicit pin), and
   `EncodeJpegV2` (explicit pin). `imageflow` translate.rs has arms for
   both.
6. Fix any tests that referenced deringing.
7. Commit, CI, self-merge, publish.

**One PR. Major version bumps on affected crates. ~1-4 hours hands-on.**

### Scenario C: narrow a range

Developer narrows `Contrast.amount` range from `-1.0..=1.0` to `-0.5..=0.5`:

1. Edit `zenfilters/src/filters/contrast/v2.rs`.
2. `cargo test` — snapshot test fails:
   *"BREAKING: zenfilters.contrast v2 param 'amount' max narrowed from
   1.0 to 0.5."*
3. Developer must revert or create a v3. The snapshot test is the firewall.

## Semver Enforcement Matrix

| Layer | Tool | Catches |
|---|---|---|
| Rust API | `cargo semver-checks` | type/fn rename, removal, signature change |
| Schema | snapshot test + `check_additive_compat` | range narrowing, param removal, variant removal |
| Full registry | workspace-level snapshot test | any dep's schema change not reflected in snapshots |
| Profile | profile validation xtask | missing nodes, inconsistent majors |
| Wire format | golden serialization test | JSON shape changes without version bump |
| Publish | preflight check | unbumped versions on changed crates |
| Publish | clean-room verify | stale or missing publishes |

## Resolved Decisions

- **Repository layout**: monorepo for `zenpipe` + `zenfilters` +
  `zencodecs` + `imageflow-api` + `imageflow` + `imageflow-abi` +
  `zen-codegen`. Other zen crates stay separate.
- **No `in-process` feature flag**: the bundle crate (`imageflow`) provides
  the executor. `imageflow-api` has no optional deps on zen crates.
- **Executor injection**: `imageflow-api` defines the `Executor` trait.
  `imageflow` provides `InProcessExecutor`. `imageflow-abi` uses
  `imageflow::Engine` which bundles both.
- **Translation layer location**: generated `translate.rs` in the
  `imageflow` bundle crate, not in `imageflow-api` or `zenpipe`.
- **Wire type generation, not re-export**: `imageflow-api` generates its
  own standalone types from schemas. Typed enums replace string params.
  Validating parsers replace raw serde.
- **Schema-driven validation**: generated `parse()` functions enforce
  ranges, enum validity, and unknown-field policy. `ValidationMode`
  (`Strict` / `Lenient` / `LegacyV2`) is set per endpoint.
- **Version mismatch detection**: Strict-mode validation is the primary
  correctness guarantee ("does it parse"). `imageflow_abi_profile_version()`
  provides early-warning startup checks for language wrappers.
- **Publish safety**: xtask preflight checks (unbumped version detection)
  + post-publish clean-room verify (builds against crates.io in isolation).
- **Deprecation policy**: all node majors stay compiled in forever by
  default (image processing needs reproducibility). Opt-out via crate
  features if binary size matters.

## Open Questions

- **Compile-time enforcement**: whether snapshot checking should eventually
  move from integration tests into the derive macro itself (reading the
  snapshot file at compile time). Tests are sufficient to start.
- **Profile cadence and naming**: date-based (`zen-2026-q2`) vs
  semver-like (`zen-v3.1`).
- **Implementation version pinning**: whether to expose per-node
  implementation version pinning as a user-facing feature.
- **Eventual rename**: whether `zenpipe` becomes `imageflow-core` after
  full C migration and feature parity. The monorepo layout supports this
  as a crate rename without repo restructuring.
